# CLAUDE.md — BE Learning Platform

## Project Overview

A learning platform that renders ~120 markdown files (BE Learning Roadmap) into a professional interactive web app. GitHub Dark / Terminal aesthetic. No backend — static site deployed on GitHub Pages/GitLab Pages.

## Architecture

- **Roadmap content**: Markdown files in `level-*/` directories (source of truth, DO NOT modify)
- **Platform app**: React SPA in `be-learning-platform/` directory
- **Content pipeline**: `scripts/build-manifest.ts` scans `level-*/` dirs → generates `public/content/manifest.json` + copies `.md` files to `public/content/`
- **Design spec**: `docs/superpowers/specs/2026-03-18-be-learning-platform-design.md`
- **Implementation plan**: `docs/superpowers/plans/2026-03-18-be-learning-platform.md`

## Coding Standards

### TypeScript
- Strict mode enabled
- All components use `.tsx` extension
- Types defined in `src/types.ts` — import from there, don't duplicate
- Prefer `interface` over `type` for object shapes
- No `any` — use proper types or `unknown`

### React
- Functional components only
- Custom hooks in `src/hooks/` — prefix with `use`
- Keep components focused: one file, one responsibility
- Co-locate component-specific logic; shared logic goes to `hooks/` or `lib/`

### Styling
- Tailwind CSS utility classes — no inline styles, no CSS modules
- Theme tokens via CSS custom properties mapped to `gh-*` Tailwind classes
- Dark mode is default. Light mode via `.light` class on `<html>`
- Color tokens: `gh-bg-primary`, `gh-bg-secondary`, `gh-bg-nav`, `gh-border`, `gh-text-primary`, `gh-text-secondary`, `gh-accent-green/blue/orange/red/purple`
- Responsive: mobile-first. Breakpoints: `md:` (768px), `lg:` (1024px)

### File Organization
```
src/
├── components/
│   ├── ui/          ← Reusable primitives (Badge, ProgressBar)
│   ├── layout/      ← App shell (TopNav, Breadcrumb, BottomNav, Sidebar)
│   ├── dashboard/   ← Dashboard-specific components
│   ├── content/     ← Markdown rendering components
│   └── features/    ← Feature components (Search, Checklist, ThemeToggle)
├── hooks/           ← Custom React hooks
├── lib/             ← Non-React utilities (storage, manifest, search-index)
├── pages/           ← Route-level components (Dashboard, Lesson)
├── styles/          ← Global CSS
└── types.ts         ← Shared TypeScript interfaces
```

### Naming Conventions
- Components: PascalCase (`LevelCard.tsx`)
- Hooks: camelCase with `use` prefix (`useProgress.ts`)
- Utilities: camelCase (`storage.ts`)
- CSS classes: Tailwind utilities only

## Content Structure (Important)

The markdown content has **inconsistent nesting depth**:

- Level 1: `level-1/topic/lesson.md` (2 levels)
- Level 2: `level-2/topic/subtopic/lesson.md` (3 levels deep — e.g., `02-mongodb/03-schema/01-basic-types.md`)
- Level 5: `level-5/lesson.md` (flat, no topics)
- Some levels have no content yet (empty dirs → `hasContent: false`)
- Special files: `checklist.md`, `README.md`, `cheatsheet.md`, `exercises/` — handled differently by manifest builder

The manifest builder normalizes all of this. Read the spec for full normalization rules.


## Key Decisions

- **No backend** — all state in localStorage, content bundled at build time
- **Runtime markdown rendering** — `.md` files fetched and rendered via `react-markdown`, not pre-compiled
- **Per-level checklists** — checklist items come from `checklist.md` per level, NOT per lesson
- **Cosmetic level locking** — all levels accessible, "locked" is just visual dimming
- **SPA routing** — uses `404.html` redirect trick for GitHub Pages compatibility
- **Slugs** — generated by stripping numeric prefixes and `.md` extension (`01-folder-overview.md` → `folder-overview`)

## Working Rules (MUST follow)

### 1. Plan Mode Default
- Vào plan mode cho mọi task phức tạp (3+ bước hoặc quyết định kiến trúc)
- Nếu gặp lỗi → DỪNG và re-plan ngay, đừng cố đẩy tiếp
- Dùng plan mode để verify, không chỉ để build
- Viết spec chi tiết trước để giảm mơ hồ

### 2. Subagent Strategy
- Dùng subagent thường xuyên để giữ context window sạch
- Offload research, exploration, phân tích song song cho subagent
- Vấn đề phức tạp → ném nhiều compute qua subagent
- Mỗi subagent chỉ 1 task

### 3. Self-Improvement Loop
- Sau mỗi lần user sửa → cập nhật tasks/lessons.md
- Viết rule để không lặp lại lỗi cũ
- Ruthlessly iterate cho đến khi giảm tỷ lệ sai
- Review lessons đầu mỗi session

### 4. Verification Before Done
- Không bao giờ đánh dấu hoàn thành mà chưa chứng minh nó chạy
- Diff behavior giữa main và changes
- Tự hỏi: "Staff engineer có approve cái này không?"
- Chạy test, check log, demo tính đúng đắn

### 5. Demand Elegance (Balanced)
- Với thay đổi phức tạp, hỏi: "Có cách nào elegant hơn không?"
- Nếu fix có vẻ hacky → "Biết tất cả những gì tôi biết, implement cách elegant"
- Skip cho fix đơn giản — đừng over-engineer

### 6. Autonomous Bug Fixing
- Nhận bug report → just fix it
- Dùng logs, errors, failing tests để chẩn đoán
- Không cần user context switching
- Tự fix failing CI tests

## Don'ts

- Do NOT modify files in `level-*/` directories — those are content source files
- Do NOT use inline styles — use Tailwind classes
- Do NOT add a backend or API — this is a static site
- Do NOT use `localStorage` directly — use `src/lib/storage.ts` wrapper
- Do NOT hardcode content structure — always derive from manifest
